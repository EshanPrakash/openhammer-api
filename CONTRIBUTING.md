# Contributing to OpenHammer API

How this project is put together, and how to get a change tested and submitted.

---

## Quick Start

```bash
git clone https://github.com/EshanPrakash/openhammer-api.git
cd openhammer-api
python -m venv venv
source venv/bin/activate
pip install -r requirements-dev.txt   # includes requirements.txt + pytest
uvicorn api.main:app --reload
```

Visit `http://localhost:8000/docs` for the interactive API docs.

### MCP Server

`api/mcp_server.py` needs a separate venv — `mcp` requires newer starlette/anyio than the pinned `fastapi==0.104.1` supports, so it can't share the main venv:

```bash
python3 -m venv venv-mcp
source venv-mcp/bin/activate
pip install -r requirements-mcp-dev.txt   # includes requirements-mcp.txt + pytest + pytest-asyncio
python -m api.mcp_server
```

## Project Structure

```
api/
  main.py               # app setup, middleware, root/stats/factions/units routes
  models.py             # Pydantic schemas (Unit, FactionInfo, StatsResponse, ...)
  data_loader.py        # loads data/json/{edition}/*.json into memory at startup
  routers/              # units, factions, weapons, abilities, bulk endpoints
  mcp_server.py         # MCP tools over the same data_store, own venv/deploy (see below)
scripts/
  sync_BSData.py        # clones/pulls the upstream BSData source repos
  universal_parser.py   # parses raw catalogue files into data/json/{edition}/
data/
  json/{edition}/       # generated unit data, one JSON file per faction (tracked in git)
  BSData-10e/           # raw XML catalogues from BSData/wh40k-10e (gitignored, generated by sync script)
  BSData-11e/           # raw JSON catalogues from BSData/wh40k-11e (gitignored, generated by sync script)
tests/
  test_api.py           # pytest suite, runs in-process, no live server needed
tests_mcp/
  test_mcp_server.py    # pytest suite for the MCP server, own venv (see below)
```

The API layer (`api/`) never talks to BSData directly — it only ever reads `data/json/{edition}/`. That split is deliberate: the data pipeline (`scripts/`) is an offline, occasionally-run process (also runs monthly via `.github/workflows/sync-data.yml`), while the API is a fast, in-memory read layer over whatever's already been generated.

`api/mcp_server.py` is deployed and tested separately from the rest of `api/` — see `requirements-mcp.txt` and `Dockerfile.mcp` (deploy) and `requirements-mcp-dev.txt`/`pytest-mcp.ini` (tests) — because the `mcp` package's dependency versions conflict with the main app's pins. It imports `data_store` directly from `api/data_loader.py` rather than calling the REST API over HTTP, so filter/sort logic (e.g. `sort_units`) lives once in `data_loader.py` and both `api/main.py` and `api/mcp_server.py` call it — if you change search/filter/sort behavior, check both call sites.

## Adding a New Edition

Routes are namespaced `/v1/{edition}/...` and `DataStore.load_all()` in `api/data_loader.py` automatically picks up every subdirectory under `data/json/`. **Adding a new edition never requires touching `api/`.** All the work lives in `scripts/`:

1. BSData ships each edition's catalogues in whatever format that edition's tooling uses — 10th Edition is XML (`.cat` files, parsed by `parse_catalogue()` using `xml.etree.ElementTree`), 11th Edition is JSON (parsed by `parse_catalogue_json()`). A future edition might use either, or something new — check the upstream `BSData/wh40k-<edition>` repo first.
2. Both parser functions produce the same `unit_data` dict shape, matching the `Unit` model in `api/models.py`. If you're adding a new format, mirror whichever existing parser is closer and reuse `clean_text()`/`extract_faction()`/`get_output_filename()` rather than duplicating them.
3. Add `(repo_url, local_path)` to `scripts/sync_BSData.py`'s `REPOS` list, and a `(filepath, faction_type, parent_catalogue)` list for the new edition in `scripts/universal_parser.py`'s `main()`, following the existing 10e/11e lists as a template.
4. Add the new local data directory (`data/BSData-<edition>/`) to both `.gitignore` and `.dockerignore`.
5. Run `python scripts/sync_BSData.py && python scripts/universal_parser.py`, sanity-check the output (unit/faction counts, spot-check a few known units), then start the API locally and hit a few `/v1/{edition}/...` endpoints before committing.

Some BSData catalogue files are "wrapper" files that reference a separate "Library" file holding the actual roster (or vice versa — verify per file, don't assume the pattern is consistent across editions). Check `sharedSelectionEntries` count or unit count in the output before assuming which file is the real data source.

## Running Tests

```bash
pytest
```

The suite (`tests/test_api.py`) runs entirely in-process via FastAPI's `TestClient` — no live server or network access needed. A few conventions to follow if you're adding tests:

- **No hardcoded test data**: `data/json/{edition}/` is regenerated monthly from upstream BSData, and specific units get added, renamed, rebalanced, or removed by GW. Fetch a real sample via the API itself (see the `sample_unit`/`sample_faction` fixtures) and assert on structural/contract properties instead.
- **Watch endpoint `limit` bounds**: several endpoints have low default or max `limit` values relative to the real data (e.g. `/abilities/keywords/list` caps at 500 but there are 1,300+ unique keywords) — a test that doesn't account for this will pass or fail for the wrong reason.
- **Prefer behavior over counts**: a filter narrows results, sorting is applied correctly, a 404 comes back for a bad edition/id — not exact counts that'll drift with the next data sync.

If you changed `api/mcp_server.py`, also run the MCP suite (own venv, since it needs the `mcp` package):

```bash
source venv-mcp/bin/activate
pip install -r requirements-mcp-dev.txt
pytest -c pytest-mcp.ini
```

Same conventions apply — `tests_mcp/test_mcp_server.py` avoids hardcoding unit/faction/weapon/ability names for the same reason.

## Submitting a Change

1. Run `pytest` and make sure it passes.
2. If you changed `api/mcp_server.py`, also run the MCP suite (`pytest -c pytest-mcp.ini` from `venv-mcp/`) and make sure it passes.
3. If you changed `scripts/universal_parser.py` or added an edition, run the pipeline and spot-check the generated `data/json/` output before committing it — don't commit generated data you haven't looked at.
4. If you changed a response shape or added an endpoint, update the README (`API Endpoints` and `JSON Structure` sections as relevant).
5. Keep commits scoped — a data-pipeline change and an unrelated test/doc change should usually be separate commits.
6. Open a PR describing what changed and why.

## Reporting Issues

- **Wrong or missing unit data**: usually an upstream BSData issue, not this repo's — check [BSData/wh40k-10e](https://github.com/BSData/wh40k-10e) or [BSData/wh40k-11e](https://github.com/BSData/wh40k-11e) first. If the data in BSData is correct but this API is misrepresenting it, that's a parser bug worth filing here.
- **API bugs**: wrong filter behavior, broken endpoint, bad response shape — file an issue with the request you made and what you expected vs. got.
- **Feature requests**: new endpoints, filters, or a new edition are welcome as issues too.
